Email System Overview
The email system uses SendGrid to send transactional emails at key points in the application workflow. There are 4 email types, all defined in server/email.ts.

Files Involved
1. server/email.ts — The Email Module
This is the core file. It exports 4 functions:

sendDoctorApprovalEmail() — Sent to the assigned doctor when an application needs review. Contains:

Patient name, email, package name, application ID
A summary of key form fields only (disability condition, state, medical condition, reason — not the full dump)
A green "Review & Approve" button linking to /review/{token}
Blue header bar with "Handicap Permit Services" branding
sendAdminNotificationEmail() — Sent to the admin notification email at the same time. Same structure but with a purple header and shows which doctor was assigned. Also has the "Review & Approve" button so admin can approve directly too.

sendPatientApprovalEmail() — Sent to the patient after the doctor approves. Green header, congratulatory message, and a "View My Dashboard" button linking to their documents page.

sendDoctorCompletionCopyEmail() — Sent to the doctor when auto-complete is ON. Teal header, tells them the patient was auto-sent their document, includes summary for their records.

Key design decision — Summary vs Full form data:
The emails used to dump every single form field (20+ fields including name, address, zip code, etc.) into the email body. That made the emails way too long. Now there's a formatFormDataSummary() function that only shows these specific keys:

disabilityCondition, disability_condition, medicalCondition, medical_condition,
state, duration, duration_requested, durationRequested, permitType, permit_type, reason
It does case-insensitive matching against the form data keys, so it works regardless of whether the field names use camelCase or snake_case. If none of those keys exist in the form data, the "Application Details" section is hidden entirely.

The old formatFormDataFull() function still exists in the file if you ever need a full dump for debugging or a different email type.

2. server/routes.ts — Where Emails Get Triggered
Emails are triggered in these places:

After payment / send-to-doctor (around line 2085+):
When an application is sent to a doctor (either manually by admin or automatically after payment), the code:

Creates a review token (random 32-byte hex string) stored in Firestore doctorReviewTokens collection
Sets 7-day expiration on the token
Updates application status to doctor_review
Builds the review URL: https://{host}/review/{token}
Calls sendDoctorApprovalEmail() with the doctor's contact email
Checks adminSettings.notificationEmail — if set, calls sendAdminNotificationEmail() with that address
Both emails fire in parallel (non-blocking .catch())
After doctor approval (in POST /api/review/:token/decision):
When a doctor clicks Approve on the review portal:

Updates the token status to approved
Updates the application status to doctor_approved
Calls autoGenerateDocument() to create the permit
Calls sendPatientApprovalEmail() with a link to their dashboard
The test endpoint (POST /api/admin/test-review-email):
This is an admin-only endpoint that lets you send a test review email to any address using a real application's data. It creates a real working token so the "Review & Approve" button actually loads the portal. Parameters: { applicationId, recipientEmail }.

3. client/src/pages/DoctorReviewPortal.tsx — The Review Page
This is the page doctors land on when they click "Review & Approve" in the email. It's a public route at /review/:token — no login required.

Fetches application data via GET /api/review/:token
Shows patient info, application details, and all form data
Doctor types optional notes and clicks Approve or Deny
Submits via POST /api/review/:token/decision
Shows confirmation screen after submission
4. Environment Variables Needed
SENDGRID_API_KEY — Your SendGrid API key
SENDGRID_FROM_EMAIL — The verified sender email (falls back to noreply@parkingrx.com)
The admin notification email is stored in Firestore under adminSettings/default → notificationEmail field, configurable from the Settings page
5. Firestore Collections Used
doctorReviewTokens — Stores review tokens with fields: id, applicationId, doctorId, token, status (pending/approved/denied), expiresAt, createdAt, usedAt
adminSettings — Document default has notificationEmail field
